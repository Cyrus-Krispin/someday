## Problem Statement

Modern friend groups live in different timezones and have conflicting schedules. Coordinating simultaneous multiplayer gaming is nearly impossible. The result: friends who want to play together can't find a time that works.

This game solves that by making multiplayer asynchronous—you play when it's your turn, at your own pace. A group of friends builds a persistent world together over weeks or months, creating shared memories and progression, without ever needing to be online at the same time.

---

## What Makes This Cool

**Friends building a persistent world together, on their own schedules, across timezones.**

- You see what your friends built, react to it, expand on it—on your schedule
- The world evolves as a shared creation with a real winner at the end
- Months-long campaign with friends feels natural and low-friction
- Spectating and chat let you stay socially connected even off-turn
- Every server is a unique world—different terrain, different resource distribution, different story

## Premises

1. **One world MVP:** One procedurally generated world, full mechanics working, no multi-server yet
2. **Sequential turns:** Players take turns in order—Player 1 → 2 → 3 → back to 1. No simultaneous action conflicts by design
3. **Turn budget — movement + actions are separate:**
   - **Movement budget: 6 points per turn.** Terrain costs: Grassland 1pt/tile · Forest 2pt/tile · Mountain 3pt/tile · Desert 2pt/tile · Water impassable. Unspent movement points do not carry over.
   - **Action budget: 2 actions per turn.** Actions: Gather (collect resources from current tile), Work (use a pre-built location — Mine, Farm, Market). Must be on the tile to use it. Actions and movement are independent — spend them in any order.
4. **Starting conditions:** Each player begins with **20 tokens** and no resources. Spawn position is evenly distributed across the map (not all in one corner).
5. **Game ends at day 30:** After 30 game-days, highest score wins. Each complete round of turns = 1 game-day
6. **24-hour turn timeout:** If a player doesn't act within 24 hours, their turn auto-skips. Notifications deferred to v2 — players check manually for MVP
7. **Turns are short sessions:** Players open the app, spend movement + actions (2-5 minutes), close it. Not a sit-down gaming session
7. **Architecture is cross-platform from day 1:** Expo + React Native Web. Same codebase runs in browser now, ships to iOS/Android later via EAS Build

## Game Design

### The World

**50×50 tile map**, procedurally generated from a seed (different seed per server). Five terrain types:

| Terrain | Move Cost | Resources | Respawn |
|---|---|---|---|
| 🌿 Grassland | 1pt/tile | Crops | 2 days |
| 🌲 Forest | 2pt/tile | Lumber | 5 days |
| ⛰️ Mountain | 3pt/tile | Ore, Gems | 7 days (Ore) · 14 days (Gems) |
| 🌊 Water | Impassable | Fish (adjacent tile action) | 3 days |
| 🏜️ Desert | 2pt/tile | Rare Gems | 14 days |

Resources deplete when gathered and respawn automatically after the listed number of game-days. Multiple players can harvest the same location on the same turn — turns are sequential so whoever goes first gets the current yield; if depleted, later players get nothing until respawn.

### Pre-Built Locations (spawn on generation, shared by all)

| Location | What you do there | Score impact |
|---|---|---|
| 🏪 Market | Buy/sell resources at prices that shift each game-day | Trade profits → score |
| ⛏️ Mine | Work a shift → earn ore + small gem chance | Resources → score |
| 🌾 Farm | Work a shift → earn crops | Resources → score |
| 🍺 Tavern | Spend tokens → +1 action next turn | **v2** |
| 🎰 Casino | Gamble tokens — 40% win, 60% lose, 2x payout | High variance score |
| 🏛️ Town Hall | View leaderboard, current world events, player locations | Info only |

### Player-Built Structures (on owned tiles)

| Structure | Cost | Effect |
|---|---|---|
| 🏠 Cabin | 10 wood | +2 tokens passive income per game-day (credited at end of game-day, to owner regardless of activity) |
| 🛒 Stall | 10 wood + 5 ore | **v2** — defer from MVP |

### Score Formula

```
Score = (Resources gathered × resource value) + (Trade profits) + (Passive income earned in tokens)
```

Tokens is the universal score unit — 1 tokens = 1 point. Everything converts to or through tokens.

Resource values (when sold at Market or counted at game end):
- Crops: 1 tokens each
- Fish: 2 tokens each
- Lumber: 2 tokens each
- Ore: 3 tokens each
- Gems: 10 tokens each

Casino and Stall are **v2 features** — excluded from MVP score formula and build plan.

### World Events (one active at a time, rotates every 5 game-days, announced at Town Hall)

Events last exactly 5 game-days each. When one ends, the next begins immediately (no gap). Cycle order is random but seeded so it's the same for all players in the same world.

| Event | Effect |
|---|---|
| 🌧️ Drought | Farms produce half this cycle |
| ⚡ Token Rush | Mines triple output this cycle |
| 🎉 Festival | Market prices double this cycle (buy cheap, sell high) |

## Turn State Machine

```
WAITING_FOR_PLAYER
  → Player opens app, sees it's their turn
  → Player spends movement points (up to 6) and actions (up to 2)
  → Player ends turn (or 24h auto-skip fires)
PROCESSING_TURN
  → Server applies actions, updates world state
  → Server advances game-day counter if all players have acted
  → Server notifies next player
WAITING_FOR_PLAYER (next player)
```

**Edge cases:**
- **Player skipped by timeout:** Turn recorded as "passed," game continues
- **Tile conflict:** Turns are sequential, so no two players act simultaneously. No conflict possible
- **Player drops out:** If a player passes 3 consecutive turns, their account goes dormant (still on leaderboard, still earns Cabin passive income — it accumulates in their balance if they return)
- **Game over trigger:** At end of game-day 30, server calculates final scores and marks game COMPLETED

## Authentication

Simple email + password signup. No OAuth for MVP—reduces complexity. Players create an account, get a join code for their world, share it with friends. Friends enter the code to join.

## Technical Architecture

### Stack

| Layer | Tech | Why |
|---|---|---|
| Frontend | Expo + React Native (web via react-native-web) | Cross-platform from day 1, you know RN |
| Rendering | **Grid of styled View components** (not Canvas) | Works on web + native without fragile bindings |
| Real-time | Socket.io client (works standard in Expo) | Turn notifications, live world updates |
| Backend | Node.js + Express + Socket.io | Fast to stand up, handles WebSocket natively |
| Database | PostgreSQL | Persistent world state |
| Hosting | Railway (backend) + Vercel (web frontend) | Free tier, no cold-start issues |

**Why grid Views, not Canvas:** Canvas/Skia bindings in React Native Web are still fragile. A tile-based game renders perfectly as a grid of styled `<View>` components—render only the 15×15 viewport around the player, update on state change. Simpler, faster to build, works everywhere. Canvas adds nothing at MVP scale.

### Data Model (core tables)

```
worlds       — id, seed, game_day, event_id, status, created_at
players      — id, email, password_hash, world_id, x, y, tokens, score, movement_remaining, actions_remaining
tiles        — world_id, x, y, terrain_type, resource_type, owner_id, structure_type
turns        — id, world_id, player_id, actions_json, created_at, auto_skipped
resources    — player_id, type, quantity
events       — id, world_id, event_type, starts_day, ends_day
```

### Real-time Flow

1. Player opens app → REST call to load world state
2. Socket.io connection opened → listen for `turn_start` event
3. Player takes actions → REST calls to `/action` endpoint
4. Player ends turn → REST call to `/turn/end`
5. Server processes, advances state, emits `turn_start` to next player's socket

## Approaches Considered

### Approach A: React Web MVP → React Native Later
- Fast to start, familiar
- Two codebases when expanding to native
- ❌ Not chosen — duplication costs too high

### Approach B: Expo + React Native Web (Day 1 Cross-Platform) ✓ CHOSEN
- Same code runs on web (hackathon) and native (expansion)
- Grid View rendering avoids Canvas fragility
- Socket.io works in Expo out of the box
- EAS Build handles iOS/Android distribution later

### Approach C: Godot
- Best game engine, true cross-platform exports
- ❌ Not chosen — different language, loses RN familiarity

## Open Questions (to resolve during build, not blockers)

1. **Viewport navigation:** Drag to pan? Arrow keys? Minimap? Decide during build
2. **Push notifications:** Deferred to v2. Players check the app manually for MVP.
3. **Casino math:** 40/60 win/loss, 2× payout gives house edge ~20%. Tune if needed
4. **World size:** 50×50 may be too large for 3 players—consider 30×30 for hackathon
5. **Socket.io reconnection:** Railway free tier may drop idle WebSocket connections. Enable `reconnection: true, reconnectionAttempts: 5` and re-emit current turn state on reconnect so clients recover without a full page reload

## Build Sequence (Day by Day)

**Day 1 — Backend + World**
1. Init Node.js + Express + Socket.io + PostgreSQL
2. World generation: use `simplex-noise` npm package, seed-based terrain, deterministic location placement
3. Data model + migrations (worlds, players, tiles, turns, resources)
4. Auth endpoints (signup, login, join world via code)
5. Turn system: advance turn, auto-skip timer, notify next player

**Day 2 — Frontend + Game Loop**
1. Init Expo + react-native-web
2. Tile grid renderer (15×15 viewport, pan to move)
3. Player movement (6 movement points per turn, terrain cost rules apply)
4. Gather action (visit Mine → earn ore, visit Farm → earn crops)
5. Market trading (buy/sell resources, price updates)
6. Score display at Town Hall

**Day 3 — Polish + Launch**
1. World events (drought, tokens rush, festival) — server-side logic
2. Game-over trigger at day 30, winner screen
3. Responsive layout (mobile browser)
4. Deploy backend to Railway, frontend to Vercel
5. Play a full game with 3 players, fix bugs
6. Launch

## Development Methodology

**Test-Driven Development (TDD) for Phase 2+**

Starting from Day 2 (Frontend + Game Loop), all implementation MUST follow TDD:

1. **Write tests first** — Define expected behavior before writing code
2. **Backend tests** — Use Jest + Supertest for API endpoints
3. **Frontend tests** — Use Jest + React Native Testing Library
4. **Test types** — Unit tests (services/utils), integration tests (API routes), component tests (UI)
5. **Red-Green-Refactor cycle** — Write failing test → Implement minimal code → Refactor

**Testing Stack:**
- Jest (test runner)
- Supertest (HTTP assertions for Express)
- React Native Testing Library (component testing)
- jest-expo (for Expo projects)

**Commit policy:** Every feature must have passing tests before commit. No exceptions.
